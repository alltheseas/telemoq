Why MoQ Keeps Your Robot Under Control

The problem: Build highly optimized, low-latency, reliable data streaming systems over unreliable transports in real-world conditions. Ship hulls, gantry cranes, and welding EMI create 10-20% packet loss on industrial WiFi (NIST).

The industry default: WebRTC — used by Polymath/LiveKit, Transitive Robotics, and Viam (gRPC+WebRTC). WebRTC's trade-off: all streams share a single ordered channel, so a lost video packet can stall control data (head-of-line blocking).

On the robot: ROS2 / DDS handles local inter-process communication at 1kHz. LCM for lightweight messaging. But none of these solve the WAN teleop transport problem — getting data reliably from operator to robot over WiFi and cellular.

MoQ (Media over QUIC): Per-stream priority scheduling over a single QUIC connection. Video degrades gracefully while safety-critical control data keeps flowing. A single protocol for the WAN teleop link.

Key Takeaway

Under congestion, MoQ's priority scheduler protects all safety and control tracks (heartbeat, SE3 commands, joints, force/torque) by starving low-priority video first. Measured: 17.4 control msgs/s.

Without priority (DDS-like), all tracks degrade unpredictably. Control throughput drops to 5.3 msgs/s — 70% less. IMU and streaming go to zero. Which tracks survive is arbitrary.

Single-stream (WebRTC-like) causes near-total blackout. Control throughput: 0.1 msgs/s — 99.4% loss. Head-of-line blocking kills everything.

The transport layer decides what gets through. With MoQ, that decision is priority-aware. With DDS, it's best-effort. With WebRTC, it's head-of-line blocked.

What happens under 15% packet loss

WebRTC: Head-of-Line Blocking

DTLS runs over a single ordered byte stream. When a video keyframe packet is lost, every stream behind it stalls — including safety-critical e-stop and joint commands. The operator loses control of the robot for 200-800ms per event. At 15% loss (NIST industrial WiFi), this happens multiple times per second.

MoQ: Priority Stream Scheduling

QUIC provides independent streams with per-track priority. Quinn's scheduler uses a BinaryHeap — strict priority starvation. Under congestion, high-priority control data (P0-P3) is sent first. Low-priority video (P20) is starved last. The robot's onboard controller uses 12ms forward extrapolation (IHMC KST pattern) to ride through brief gaps — MoQ keeps those gaps under 12ms even at 15% loss.

20% Packet Loss: Shipyards as Faraday Cages

Steel ship hulls act as Faraday cages. Gantry cranes cause multipath interference. Arc welding generates broadband EMI across the ISM band. NIST measurements show 10-20% packet loss in these environments. A teleoperated humanoid holding a welding torch cannot tolerate 200ms control gaps — e-stop must always get through.

The Problem: Reliable Streaming Over Unreliable Transports

ROS2/DDS runs on the robot at 1kHz. LCM handles lightweight messaging. But the WAN teleop link — WiFi and cellular from operator to robot — needs “optimized, reliable data streaming protocols functioning over WiFi and cellular networks.” Current solutions: Polymath uses LiveKit/WebRTC, Transitive uses WebRTC P2P, Viam uses gRPC+WebRTC. All inherit WebRTC's head-of-line blocking trade-off.

Built on IHMC's Lessons

click to expand

IHMC's 20+ years of humanoid teleoperation research — from Atlas at the DRC to Nadia and Alex — shaped how we think about priority-based communication. The DRC's degraded-network constraints informed MoQ's track hierarchy.

Safety Can't Wait Behind Video

If the network drops, the robot must stop safely. With WebRTC, a large lost video frame blocks the emergency stop from getting through. MoQ guarantees safety signals always ship first — video can wait, the e-stop cannot.

IHMC Safety • Coactive Design

Control Commands Are Tiny

At the DARPA Robotics Challenge, IHMC's Atlas operated on a 9,600 bps control link. Control commands are tiny — the bandwidth bottleneck is always video and sensor data. So when bandwidth gets tight, sacrifice the big stuff (video) to protect the small stuff (commands).

DRC Network Degradation • Team IHMC Lessons

High-Level Goals, Not Raw Joint Commands

The robot runs its own fast control loop locally. The operator sends high-level goals ("move hand here"), not low-level motor commands. This means the teleop link can tolerate brief interruptions — the robot keeps executing while waiting for the next goal.

Known Demo Limitations

This demo proves MoQ's priority scheduling mechanism. It does not replicate production teleop conditions. We document these gaps honestly so practitioners can evaluate what's demonstrated vs. what's claimed.

LimitationWhat the demo doesWhat production requires
Localhost only Pub & sub on same machine via loopback. Clocks perfectly synced. Multi-hop relay, NTP drift (1-10ms), asymmetric paths. Add ~20ms encode + ~5ms decode.
Bandwidth constraint, not random loss Bandwidth throttling triggers priority starvation. Shipyards have random packet loss (NIST: 10-20%). BBR can collapse under >5% random loss — priority scheduling doesn't help if BBR shuts down the pipe.
No real video codec 50KB zero-filled blobs at 30fps. H.264/H.265 with I-frame/P-frame dependencies. Losing an I-frame degrades more than a P-frame. No GOP structure tested.
No bidirectional flow Single publisher simulates both directions. Two independent publishers at relay. Bidirectional priority under asymmetric congestion is unsolved.
Forward extrapolation claimed, not shown Inter-arrival gaps displayed vs. 12ms KST window. Simulated robot arm extrapolating trajectory during gaps, then snapping back on data resume.
No connection migration Static QUIC session on loopback. Robot roaming between APs. QUIC connection IDs survive network change (vs. WebRTC ICE restart, TCP death).

The WiFi → Cellular Handoff Problem

click to expand

A humanoid robot in a shipyard moves between welding bays. Each bay has a different WiFi access point. Between bays, coverage drops and the operator's device may fall back to cellular. This is the handoff problem — and it exposes a fundamental difference between transport protocols.

ProtocolWhat happens on network changeDowntime
TCP Connection is bound to a 4-tuple (src IP, src port, dst IP, dst port). When the IP changes, the connection dies. Application must detect timeout, reconnect, re-authenticate, re-subscribe to all tracks. 2-10 seconds
WebRTC Requires ICE restart: re-gather candidates, re-negotiate connectivity checks, re-establish DTLS. Transitive Robotics and Viam both use TURN fallback, adding another hop. During ICE restart, the operator has no control of the robot. 0.5-3 seconds
DDS (RTPS) DDS discovery uses multicast on the local network segment. When the robot moves to a new subnet, discovery fails. Requires DDS router/bridge configuration per network, or WAN discovery via DDS Cloud Discovery Service. Not designed for roaming. Discovery-dependent
QUIC (MoQ) Connection identified by Connection ID, not IP/port. When the network interface changes, the client sends packets from the new address with the same Connection ID. The server validates via PATH_CHALLENGE/PATH_RESPONSE. Session continues — no re-handshake, no re-subscription. 0-RTT (one round-trip)
The shipyard scenario

Robot walks from Bay A (WiFi AP1) through a steel corridor (no coverage) to Bay B (WiFi AP2). The operator must maintain continuous control throughout. With WebRTC, the 1-3 second ICE restart in the corridor means the robot is uncontrolled while holding a welding torch. With QUIC, the heartbeat stream continues within one round-trip of reaching AP2.

The cellular fallback scenario

The operator's control station has both WiFi and 5G. WiFi drops during an AP transition. With QUIC connection migration, the session seamlessly moves to 5G — higher latency but no interruption. MoQ's priority scheduling then adapts: high-priority control (P0-P1) fits easily in cellular bandwidth, while video (P20) is starved to match the reduced pipe.

Status: QUIC connection migration is defined in RFC 9000 §9. Quinn (the Rust QUIC library MoQ uses) has client-side migration support. The moq-relay does not yet accept migrated connections. This is a protocol capability, not yet a telemoq feature.
