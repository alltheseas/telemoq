telemoq benchmark

Same code, same relay, same connection. Only variable: priority scheduling.

Methodology: moq-relay on localhost:4443 (self-signed TLS). telemoq publisher emitting 8 tracks simultaneously at aggregate ~3 MB/s — deliberately exceeding QUIC connection capacity to force congestion. No artificial packet loss needed; the scheduler must choose what to deliver.

One variable: --no-priority sets all 8 track priorities to 0 (equal). Same binary. Same relay. Same payloads. Only the QUIC stream priority header changes.

Key finding: priority scheduling provides predictable, ordered degradation.

With priority, the scheduler distributes capacity evenly across P0–P5 tracks (~2.1 msgs/s each) and completely blocks P10–P20 (video, pointcloud). Every safety-relevant track gets delivery.

Without priority, delivery becomes chaotic and unpredictable. IMU drops from 2.2/s to 0/s. Joints drops from 2.1/s to 0.4/s. Streaming drops from 2.2/s to 0.6/s. The FIFO scheduler has no concept of importance — which tracks survive is arbitrary.

Total control-track throughput (P0–P5): With priority: 12.8 msgs/s  |  Without priority: 5.7 msgs/s  |  55% loss of control data without priority.

What this means for teleop safety

With priority (MoQ default)

Every control track (P0–P5) receives ~2.1 msgs/s — evenly distributed. Heartbeat alive. Joints reporting. Force/torque contact detection active. Video and pointcloud sacrificed. The robot stays under control.

Matches IHMC DRC lesson: "9,600 bps was enough for control."

Without priority

IMU: dead (0/s). Joints: 0.4/s (80% loss vs priority mode). Streaming: 0.6/s (73% loss). Control-track throughput drops 55%. The scheduler doesn't know heartbeat matters more than video. Which tracks survive is a coin flip.

This is how WebRTC/DDS behave: all streams degrade together, unpredictably.

How it works

Priority flow: Track.priority → PriorityQueue.insert(priority, seq) → stream.set_priority() → QUIC stream header → quinn's BinaryHeap scheduler.

With --no-priority, all tracks get priority 0. The BinaryHeap orders by group sequence only (FIFO). No track is preferred. Under congestion, which tracks get capacity depends on timing, not importance.

Production context: Over WiFi/cellular with 10+ Mbps, the small control tracks (heartbeat: 170 B/s, streaming: 26 KB/s) achieve near-100% delivery while video (1.5 MB/s) degrades gracefully. This localhost stress test uses extreme congestion to make the priority mechanism clearly visible.
